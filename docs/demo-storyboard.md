# HATOBA 紹介動画 絵コンテ（v4）

> **最終成果物は発表スライド全体。** この動画はその中の **「開発した機能の紹介」パート** として1本にまとめる。
> 背景・課題・開発に至った経緯・三方よしなどの説明は **スライド側で行う**（動画には載せない）。
> 動画は **機能の紹介に専念**し、リアルタイムデモではなく編集した1本として見せる。
> （スライドへの反映依頼は [../../スライド指示_開発班から.md](../../スライド指示_開発班から.md)、技術構成・開発の進め方は [roadmap.md](./roadmap.md) を参照）

## 狙い（この動画で見せること）
**開発した機能そのもの** を一連の流れで見せる。
店員が食材を撮る → AIが食材を読み取る → AIがメニューを提案する → 学生が注文・決済する → そのメニューが実際に作れる、まで。
「何ができるアプリか」を機能ベースで伝えるのが役割。サービスの意義づけ・背景はスライドに任せる。

## 撮影方針
- **実写パート**: ①店員がスマホで食材を撮影するシーン（つかみ）、②AI提案メニューを実際に調理して完成させるシーン。
- **画面録画パート**: AI食材抽出〜メニュー提案〜公開〜学生の注文・決済はアプリ画面の録画。
- **シーン1は「実写＋画面録画」を2画面同時で見せる**: 店員が値引き品を撮影する実写と、アプリにAIが食材を読み取って登録する画面録画を**同時に並べる**（左右分割 or ワイプ）。「現実の動作」と「アプリの中の動き」を一目でつなげる。2クリップは別レイヤーに置き、開始位置・長さを揃える。
- **山場（食材写真→AIメニュー提案）は本物として動作確認済み**（AIコア2機能が実際にGeminiで動く）。
- 理想は **本物の食材をシーン2のAIに通して出てきたメニューを、シーン4でそのまま作る** こと。全シーンが1本の実話としてつながり、説得力が最大になる。

## 「提案メニュー＝実際の料理」の見せ方（シーン4）
- 買い出しのシーン自体は不要。**重要なのは「AIが出したメニュー」と「実際に作れた料理」が同じだと分かること。**
- 画面録画で出したシーン2のメニュー（料理名・AI生成画像）と、シーン4の完成皿を **画面分割 or カット切り替えで並べる**。
- テロップ例：「AIが提案 → 実際に作れた」。これで「AIの提案は絵に描いた餅じゃない」ことを証明する。

## 具体シナリオ例（同じ画を想像するための叩き台 ※実際に作る料理に合わせて差し替え）
- **シーン1で撮る余り食材**: 鶏もも肉 / キャベツ / 人参（値引きシール付き）
- **シーン2でAIが提案するメニュー例**（実際は機能②がレシピ・何食作れるか・調理時間も返す）:
  - 「鶏とキャベツの味噌だれ丼」 ¥420（アレルギー: 小麦・大豆）
  - 「鶏と野菜のあんかけ焼きそば」 ¥450（アレルギー: 小麦・卵）
  - 「チキンと春キャベツのスープごはん」 ¥380（アレルギー: 乳）
- **公開・調理するメニュー**: 「鶏とキャベツの味噌だれ丼」
- **シーン3で学生が見る表示**: 本日のメニュー「鶏とキャベツの味噌だれ丼 ¥420」／残り 8 食
- **シーン4で実際に作る料理**: 上で公開したメニューと同じもの（＝味噌だれ丼）

> 補足：シーン3の「残り食数（残数）」は、本来は機能②が出す `servings`（在庫全体で何食作れるか）から注文数を引いた値。
> デモではモック（固定データを手で減らす）で見せる。
> シーン4で実際に作る料理は、できれば本物の買い出し食材をシーン2のAIに通した結果に揃えると、全編が一貫する。

## 「本物 / モック」の切り分け
- **本物にする = シーン2（食材写真→AIメニュー提案）。** ここだけ実際にAIを動かす（実装済み）。
- 学生側の料理画像は **AIで自動生成**（機能③。公開時にサーバー側で1回生成して保持）。
- それ以外の画面（公開操作・残数・注文ボタン）はモック（固定データ・ボタン遷移）でOK。動画で動いて見えれば満点。
- 席バーコードによる注文方法は未実装のため、動画では扱わない（学生側は注文ボタンを押すところまで）。
- アレルギー・価格はシーン2の本物の出力をそのまま使うので、モック画面でも本物のデータが乗る。
