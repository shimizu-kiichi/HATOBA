# カスタマー画面の料理画像（AI生成イメージ → 実写差し替え）

> 客側（costomer）画面に出す料理写真の扱いをまとめる。
> 店が料理を完成・撮影するまでの「つなぎ」としてAI生成のイメージ画像を表示し、完成後に実写へ差し替える。
> **画像生成はサーバー側（公開API Route）で「公開時に1回だけ」行い、結果をサーバーのメモリに保存する。**
> ※ 本ドキュメントは**モック前提**。本番運用（複数端末での確実な共有・デプロイ）は今は考えない。

## 目的・背景

公開直後はまだ実際の料理写真が無い。空欄やプレースホルダーのままだと客側に魅力が伝わらないので、
**料理名・レシピから生成したAIイメージ画像を仮に表示**しておき、店が実際に作って撮影したら**実写に差し替える**。

## 採用する構成：サーバー（API）方式

画像生成は**客側フロントでは行わず、サーバーのAPI Routeの中で行う**。
店側の「公開」処理の中で画像を1回だけ生成してメモリに保存し、客側はその保存済み画像を表示するだけにする。

メリット：
- **公開時に1回だけ生成**（客が開くたびに生成しない＝効率的・安定）。
- 画像の**保存・実写差し替えが自然に**できる。
- 生成APIのキーを**サーバーの環境変数に隠せる**（安全）。

## 全体の流れ

```
[店側 store] メニュー確定（料理名・レシピ）
      │ POST（料理名・レシピ）
      ▼
[サーバー API Route]  例: POST /api/published-menu
      │ ① 料理名＋レシピからプロンプトを生成
      │ ② Pollinations.ai（リンク型・無料）で画像生成（下記「画像生成API」）
      │ ③ 返ってきた画像(base64)を保存
      │ ④ メニュー＋画像(base64)＋ isReal=false をメモリに保存（現在の1件）
      ▼
[ストア]  サーバーのメモリに「現在の1件」を保存・画像はbase64
      ▲
      │ GET（公開メニューを取得）
[客側 costomer] 保存済みのメニュー＋画像を表示
      └ isReal=false の間だけ「※イメージ画像（AI生成）」表記

——— 店が料理を完成させたら ———
[店側] 完成写真を撮影 → アップロード（POST /api/menu/photo）
      → 画像を実写に差し替え＋ isReal=true（メモリを上書き）
      ▼
[客側] 以降は実写を表示（「※イメージ画像」表記は自動で消える）
```

## 構成要素

### API Route（サーバー）
- `POST /api/published-menu` … メニュー受領 → **プロンプト生成 → 画像生成 → メモリ保存**（公開時に1回）。
- `GET  /api/published-menu` … サーバーのメモリから読み、客側へ「メニュー＋画像(base64)＋ isReal」を返す。
- `POST /api/menu/photo` … 店の**実写アップロード** → メモリの現在の1件を取得して画像を差し替え＋ `isReal=true` で上書き。

### ストア（保存先）— 採用方針：メモリに「現在の1件」だけを base64 で持つ

このサービスでは、保持すべきは **「今、公開中のメニュー」ただ1件だけ**でよい。
過去のメニューを蓄積する必要はなく、新しく公開したら**前の内容を上書き**するだけ。
モックなので**DB・ファイル・KVは使わず、サーバーのメモリ上に1件だけ持つ**方式を採用する。

- **置き場所**：サーバーのメモリ（1つの変数）。以前作った `menuStore`（globalThis に1件持つ作り）を流用する。
- **画像の持ち方**：画像は **base64（データURL）でメモリに直接持つ**。
  - 保存も差し替えも破棄も**すべてメモリ内で完結**する（`public/` にファイルを作らないので後始末が要らない）。
- **保持件数**：常に**1件のみ**。新しい公開で丸ごと上書きするため、履歴は残らない。

> 本番で「店の端末」と「客の端末」を**別インスタンス越しに確実に共有**したくなったときだけ、
> この1件を KV（Vercel KV など）に置き換える。`menuStore`（1ファイル）の中身を差し替えるだけで済む。
> **モックのうちはメモリで十分。**

### データの形（例）
```ts
type PublishedMenu = Menu & {
  image: string | null; // 画像の base64 データURL（"data:image/png;base64,..."）。未生成なら null
  isReal: boolean;      // false = AI生成イメージ, true = 店の実写
  remaining: number;    // 残数
  publishedAt: number;  // 公開時刻（任意）
};

// メモリ上に「現在の1件」だけを保持するイメージ
let current: PublishedMenu | null = null;
```

## 保存と破棄のライフサイクル（メモリ1件・base64）

メモリ上の「現在の1件」が、公開〜実写差し替え〜次の公開を通してどう変化するかを詳しく示す。

### ① 公開（新メニュー確定）
- 店側が「このメニューに決定」→ サーバーの `POST /api/published-menu` を呼ぶ。
- サーバーは料理名・レシピから**画像を1回だけ生成**し、**base64 に変換**。
- メモリの `current` を**丸ごと新しい内容で上書き**する：
  ```
  current = {
    ...メニュー(料理名・レシピ・価格・アレルゲン等),
    image: "data:image/png;base64,...(生成画像)",
    isReal: false,     // AI生成イメージ
    remaining: 作れる食数,
    publishedAt: 今,
  }
  ```
- この時点で前のメニュー・前の画像は**上書きで消える（破棄）**。

### ② 客が見る
- 客側 `GET /api/published-menu` で `current` を取得し、`image` を表示。
- `isReal === false` の間は「※イメージ画像（AI生成）」表記を出す。

### ③ 料理が完成 → 実写に差し替え（AI画像を破棄）
- 店側が完成写真を撮影 → `POST /api/menu/photo` でアップロード。
- サーバーは `current.image` を**実写の base64 で上書き**し、`current.isReal = true` にする。
  ```
  current.image  = "data:image/png;base64,...(店が撮った実写)"
  current.isReal = true
  ```
- ここで**AI生成画像はメモリ上から消える（上書き＝破棄）**。
  - base64 をメモリに持っているので、上書きした瞬間に古い画像データは参照されなくなり、
    JavaScript のガベージコレクションで自動的に破棄される（ファイル削除などの後始末は不要）。
- 客側は次の取得で実写を表示し、`isReal === true` なので**「※イメージ画像」表記は自動で消える**。

### ④ 次の公開（メニュー入れ替え）
- 再び `POST /api/published-menu` が呼ばれると、`current` が**新しいメニュー＋新しいAI画像で丸ごと上書き**される。
- これで①に戻る。常に保持しているのは**最新の1件だけ**。

### まとめ（状態遷移）
```
（空 null）
   │ 公開
   ▼
[AI画像 / isReal=false]  ──実写アップロード──▶  [実写 / isReal=true]
   ▲                                                    │
   └──────────────── 次の公開で丸ごと上書き ◀───────────┘
```
- 保持は常に1件。新公開・実写差し替えは**すべて「上書き」**で表現でき、古い画像は破棄される。
- メモリ＋base64 なので、生成・差し替え・破棄が**外部ファイルやDBなしで完結**する。

## 画像生成API（使用するサービス）

**採用：Pollinations.ai（リンク型の無料画像生成）。**

URL を1本叩くだけで画像が返る「リンク型」サービス。**APIキーの登録も課金も不要**で、モックの画像生成にちょうどよい。必ず**サーバー（API Route）から呼ぶ**（プロンプト整形・base64化をサーバーで完結させる）。

> 補足（Gemini画像を見送った理由）：当初は Gemini `gemini-2.5-flash-image`（Nano Banana）を検討したが、
> **無料枠が1日0リクエスト（`limit: 0`）で、課金を有効化しないと画像生成できない**ことが判明したため、
> 課金不要で動く Pollinations.ai に切り替えた。テキストのメニュー提案（機能②）は引き続き Gemini を使う。

### 選定の背景：リンク型（URL）と APIキー型の違い
画像生成APIには2形式あり、本プロジェクトは**リンク型**を採用した。

| | リンク型（Pollinations） | APIキー型（Gemini / OpenAI 等） |
|--|--|--|
| 呼び出し | URLを1本 `fetch` するだけ（認証なし） | キーを認証ヘッダに付けてJSONで送受信 |
| 費用・準備 | **登録も課金も不要・即利用可** | アカウント登録＋多くは課金（無料枠は極小） |
| モデル選択 | 限定的（無料枠は `sana` のみ） | 細かく固定可（モデルID・解像度・各種パラメータ） |
| 品質・安定性 | 共有基盤ゆえ不安定（遅延・混雑・たまに失敗） | 商用基盤で安定 |
| 向く用途 | **モック・デモ**（HATOBAはこれ） | 本番・商用・品質/権利が重要なケース |

→ HATOBAは**発表用デモ**なので「無料・キー不要・即動く」を優先してリンク型にした。
本番化して品質・安定性・権利の明確さが要るなら、APIキー型（モデル固定できる商用API）への移行が選択肢になる。

### 使用モデル `sana` について
- `sana` は NVIDIA（MIT Han Lab と共同）の text-to-image モデル。**深圧縮オートエンコーダ＋線形アテンション**で「軽量・高速・高解像度」に振った設計。無料の共有サービスが大量処理をさばくのに向くため、Pollinations の無料枠で採用されていると見られる。
- **品質の限界**：最上位の写実モデル（FLUX・Imagen 等）に比べると、究極の写真らしさ・細部（文字・小物・複雑な盛り付け）は崩れやすい。だからこそ**「※イメージ画像（AI生成）」の明記が一層重要**。
- Pollinations が動かす具体的な sana のバージョン・設定は非公開。中身まで固定したい場合はキー型APIに分がある。

### 画風の統一（モデル・サイズ・seed・スタイル文を固定）
メニューごとに絵がバラけないよう、次を固定して画風をそろえる。
- **モデル固定**：`model=sana`。Pollinations の無料（匿名）枠で叩けるモデルは現状 `sana` のみ（`https://image.pollinations.ai/models` で確認可）。明示しておけば将来モデルが増えても勝手にブレない。
- **サイズ固定**：`width=1024&height=768`（4:3）。
- **seed固定**：料理名から安定した整数seedを作る → **同じメニューは毎回同じ画像**になる（再公開しても絵が変わらない）。
- **共通スタイル文**：俯瞰45度・木のテーブル・自然光…等の英語スタイル文を毎回プロンプト末尾に付ける。
- 補足：本体（sanaの推論基盤）を直接叩くことは無料ではできない（各社の認証・課金が要る）。**Pollinations経由のまま `model` を固定するのが、無料・キー不要で統一する唯一現実的な方法**。

### 呼び出し方（サーバー側）
- 画像URLを組み立てて `fetch` し、返ってきたバイナリを base64 データURL に変換してメモリの `image` に保存する：
  ```ts
  const prompt = buildImagePrompt(menu);   // 料理名＋食材＋調理法＋共通スタイル文
  const seed = seedFromName(menu.menuName); // 料理名から安定seed
  const url = `https://image.pollinations.ai/prompt/${encodeURIComponent(prompt)}`
            + `?width=1024&height=768&nologo=true&model=sana&seed=${seed}`;
  const res = await fetch(url, { signal: AbortSignal.timeout(60000) });
  const buf = Buffer.from(await res.arrayBuffer());
  const mimeType = res.headers.get("content-type") ?? "image/jpeg";
  const image = `data:${mimeType};base64,${buf.toString("base64")}`;
  ```

### 実装メモ
- **キー不要**：Pollinations はURLを叩くだけ。`.env` に画像用のキーは要らない。
- 返ってくるのは画像バイナリ（通常 `image/jpeg`）。base64 データURL化してそのままメモリに保存（本ドキュメントの保存方針と一致）。
- 失敗時（タイムアウト・サービス障害等）は画像を `null` のままにして、客側はプレースホルダー表示にフォールバックする。
- テキストのメニュー提案（suggest-menu）は引き続き `@google/genai`／`GEMINI_API_KEY`。**画像だけ別サービス**になる。

## 状態の整理

| 状態 | 表示する画像 | 表記 |
|--|--|--|
| 仮（公開直後〜調理中） | AI生成イメージ | 「※イメージ画像（AI生成）」 |
| 確定（料理完成・撮影後） | 店が撮った実写 | 表記なし |

### 表示ロジック（重要）

「※イメージ画像（AI生成）」の表記は、**表示している画像がAI生成かどうかに連動**させる。
- 画像が **AI生成イメージ**（`isReal=false`）→ 表記を**出す**
- 画像が **店の実写に差し替わった**（`isReal=true`）→ 表記を**自動で消す**

つまり表記は `isReal` にひも付け、**実写を挿入した瞬間に表記も消える**実装にする。

## 技術メモ・注意点

- **画像生成はサーバーで1回**：公開時に生成してメモリに保存。客側は保存画像を表示するだけ（再生成しない）。
- **画像はキー不要・テキストは GEMINI_API_KEY**：画像生成は Pollinations.ai（キー不要）、メニュー提案は `GEMINI_API_KEY`（サーバー側 `.env.local`）。キーはクライアントに出さない。
- **保存先はメモリ**：メニューと画像(base64)をメモリの「現在の1件」に持つ。モックなのでDB・ファイル・KVは使わない。
- **「イメージ画像」明記は必須**：AI生成は実物と異なるため、誤認防止の表記を必ず入れる。
- **遅延対策**：Pollinations の画像生成には数秒〜十数秒かかる。**公開時に1回だけ生成・保存**しておけば、客側の表示は遅延の影響を受けない。失敗時は画像 `null` でプレースホルダーにフォールバック。
- **無料サービスのリスク**：Pollinations は非公式・無料の共有サービスのため、**仕様変更・モデル入れ替え・停止が予告なく起こりうる**（実際 flux→sana と変わった）。また生成画像の**商用利用・著作権の扱いが曖昧**。デモには十分だが、本番でそのまま使うのは避け、必要なら権利・SLAの明確なキー型APIへ移す前提とする。

## 決定事項（このドキュメントで確定）

- 本ドキュメントは**モック前提**。本番運用（複数端末の確実な共有・デプロイ）は今は考えない。
- 画像生成は**サーバー（公開API Route）で公開時に1回だけ**。
- 画像生成は **Pollinations.ai（リンク型・キー不要・無料）**。テキストのメニュー提案のみ `GEMINI_API_KEY` を使う（※Gemini画像は無料枠0のため見送り）。
- 保存は**サーバーのメモリに「現在の1件」だけ**（base64画像込み）。DB・ファイル・KVは使わない。
- 画像は**base64（データURL）でメモリに保持**。差し替え・破棄はすべて上書きで完結。
- 表記の出し分けは **`isReal`** に連動（実写差し替えで自動的に消える）。
